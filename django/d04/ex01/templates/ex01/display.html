  • URLs: 단일 함수를 통해 모든 URL 요청을 처리하는 것이 가능하지만, 분리된 뷰 함수를 작성하는 것이 각각의 리소스를 유지보수하기 훨씬 쉽습니다. URL mapper는 요청 URL을 기준으로 HTTP 요청을 적절한 뷰(view)로 보내주기 위해 사용됩니다. 또한 URL mapper는 URL에 나타나는 특정한 문자열이나 숫자의 패턴을 일치시켜 데이터로서 뷰 함수에 전달할 수 있습니다.
  • View: 뷰는 HTTP 요청을 수신하고 HTTP 응답을 반환하는 요청 처리 함수입니다. 뷰는 Model을 통해 요청을 충족시키는데 필요한 데이터에 접근합니다. 그리고 탬플릿에게 응답의 서식 설정을 맡깁니다.
  • Models: 모델은 응용프로그램의 데이터 구조를 정의하고 데이터베이스의 기록을 관리(추가, 수정, 삭제)하고 쿼리하는 방법을 제공하는 파이썬 객체입니다.
  • Templates: 탬플릿은 파일의 구조나 레이아웃을 정의하고(예: HTML 페이지), 실제 내용을 보여주는 데 사용되는 플레이스홀더를 가진 텍스트 파일입니다. 뷰는 HTML 탬플릿을 이용하여 동적으로 HTML 페이지를 만들고 모델에서 가져온 데이터로 채웁니다. 탬플릿으로 모든 파일의 구조를 정의할 수 있습니다.탬플릿이 꼭 HTML 타입일 필요는 없습니다!

Django애플리케이션의 레이어

  1. 미들웨어 요청
  2. URL 라우터(URL 디스패처)
  3. 견해
  4. 컨텍스트 프로세서
  5. 템플릿 렌더러
  6. 응답미들웨어

요청이 들어올 때마다 요청 미들웨어에 의해 처리됩니다. 우리는 여러 미들웨어를 가질 수 있습니다. 프로젝트 설정(settings.py)에서 찾을 수 있습니다. 요청 미들웨어는 요청을 처리하는 동안 순서를 따릅니다. A, B, C 순서로 요청 미들웨어가 있는 경우 요청이 먼저 미들웨어 A에 의해 처리된 다음 B, C에 의해 처리된다고 가정합니다. Django는 기본 미들웨어 무리를 제공합니다. 자체 또는 사용자 정의 미들웨어를 작성할 수도 있습니다. 미들웨어에 의해 처리된 요청 후 URL 라우터 또는 URL 디스패처에 제출됩니다.

URL 라우터는 요청 미들웨어의 요청을하고 그것을에서 URL 경로를 소요 요청. URL 경로 URL을 기반으로 라우터는 요청 경로를 사용 가능한 URL 패턴과 일치시키려고 시도합니다. 이러한 URL 패턴은 정규식 형식입니다. URL 경로를 사용 가능한 URL 패턴과 일치시킨 후 요청은 URL과 연결된 보기에 제출됩니다.

이제 우리는 비즈니스 로직 레이어 보기에 있습니다. Views는 요청 및 요청 데이터(GET, POST 등으로 전송된 데이터)를 사용하여 비즈니스 로직을 처리합니다. 요청이 전송된 컨텍스트 프로세서 보기에서 요청을 처리한 후 요청 컨텍스트 프로세서를 사용하여 템플릿 렌더러가 템플릿을 렌더링하여 HTTP 응답을 생성하는 데 도움이 되는 컨텍스트 데이터를 추가합니다.

다시 요청 은 이를 처리하기 위해 응답 미들웨어로 다시 보냅니다. 응답 미들웨어는 요청을 처리하고 헤더 정보/본문 정보를 추가하거나 수정한 후 클라이언트(브라우저)로 다시 보냅니다. 브라우저가 처리하고 최종 사용자에게 표시한 후.
